v0.2 adds the menu layer without turning the MVP into a full ordering system.

This visual contract keeps the QR session, chat, and staff request APIs from v0.1, then adds structured menu categories, menu items, images, prices, allergens, availability, visibility, and staff menu management.

10 Database tables in scope
29 Documented endpoints
14 Menu-specific endpoints
0 Real cart/order endpoints included

What Changed From v0.1

  • Menu is now structured Categories, items, images, prices, availability, and allergens can be stored and returned by API.
  • AI can answer menu questions It should answer only from real menu data returned by the database.
  • Ghost orders are reduced `is_visible` and `is_available` let staff hide or mark sold-out items.
  • Ordering is still out of scope Cart, order placement, kitchen display, and payment remain planned work.

Database Scope

v0.2 keeps the six v0.1 MVP tables and adds four menu tables.

AI Menu Rule

The menu API changes how the chat endpoint should answer dish questions.

Allowed

  • Recommend dishes returned from `menu_items`.
  • Mention exact `price_cents`, `currency`, availability, and allergens.
  • Say menu data is not available when the database has no matching item.
  • Suggest staff help when allergen or safety information is unclear.

Not Allowed

  • Invent dishes, prices, images, availability, or allergens.
  • Promise that food is safe for a customer without reliable allergen data.
  • Create real orders, because cart and order tables are not in v0.2.
  • Hide uncertainty when menu data is incomplete.

Say This To The Team

API Contract v0.2 expands the MVP from table communication to structured menu browsing and menu-aware AI answers. It still does not include cart, order placement, kitchen display, payment, or reviews. Those features need their own database tables and should stay planned until the team chooses that scope.
